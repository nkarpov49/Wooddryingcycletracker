# Решение ошибки "Service invoked too many times for one day: urlfetch"

## 🚨 Проблема

```
Exception: Service invoked too many times for one day: urlfetch.
```

Эта ошибка означает, что достигнут **дневной лимит запросов UrlFetch** в Google Apps Script.

**Что произошло:**
- Вы видели в логах: `{"1":59,"2":2,"3":2,...}` — проценты **неправильные**
- Это означало, что формула читалась неверно (2% вместо 200%)
- Вы много раз запускали скрипт вручную для тестирования
- Лимит в 20,000 запросов/день был исчерпан
- **Сейчас**: Лимит исчерпан, нужно подождать до завтра

---

## ⏰ ГЛАВНОЕ РЕШЕНИЕ: Подождите до завтра!

### Когда сбросятся лимиты?
- **Каждые 24 часа** по времени **Pacific Time (США)**
- Обычно это около **2-3 часов ночи** по Московскому времени
- Обычно это около **9-10 часов вечера** по Вильнюсскому времени

### Что делать прямо сейчас?
1. ✅ **Удалите ВСЕ триггеры** (Triggers → Delete All)
2. ✅ **Не запускайте скрипт вручную** больше сегодня
3. ⏰ **Подождите до завтра**
4. 🧪 **Опционально**: Используйте тестовый скрипт (см. ниже)

---

## 🧪 ВРЕМЕННОЕ РЕШЕНИЕ: Тестовый скрипт

Чтобы **проверить** что данные читаются правильно **БЕЗ отправки на сервер**:

### Шаг 1: Скопируйте тестовый скрипт
Откройте файл: `/docs/google-apps-script-TEST-ONLY.txt`

### Шаг 2: Вставьте в Apps Script
```
1. Apps Script → Создайте НОВЫЙ файл (File → New → Script file)
2. Назовите: "TestOnly"
3. Вставьте код из google-apps-script-TEST-ONLY.txt
4. Сохраните
```

### Шаг 3: Запустите тест
```
1. Выберите функцию: testProgressReadingOnly
2. Нажмите Run (▶️)
3. Проверьте логи (View → Logs)
```

### Шаг 4: Проверьте результаты
```
✅ ХОРОШО (правильно):
{"1":59.2,"2":23.4,"3":115.5,...}

❌ ПЛОХО (неправильно):
{"1":59,"2":2,"3":2,...}
         ↑ЭТО НЕВЕРНО!
```

**Этот тест НЕ использует urlfetch**, так что он сработает даже при исчерпанном лимите!

## ✅ Что было исправлено

### 1. **Правильное чтение процентов**
```javascript
// БЫЛО (неправильно):
if (percentValue > 0 && percentValue <= 1) {
  percentValue = percentValue * 100;  // Работало не для всех ячеек
}

// СТАЛО (правильно):
if (typeof percentValue === 'number') {
  percentValue = percentValue * 100;  // ВСЕГДА умножаем, если число
}
```

**Теперь:**
- Формула `=(now()-$C$6)/((vlookup(...))` возвращает `0.592` → **59.2%** ✅
- Поддержка трёхзначных: `1.155` → **115.5%** ✅
- Округление до 0.1: `59.234` → **59.2%** ✅

## 📊 Лимиты Google Apps Script (бесплатный аккаунт)

| Сервис | Лимит в день |
|--------|--------------|
| URL Fetch calls | **20,000 запросов** |
| URL Fetch data received | 100 MB |
| Script runtime | 6 минут на выполнение |

---

## 🔍 Причины превышения лимита

### 1. **Слишком много триггеров**
Возможно, у вас настроено несколько триггеров, которые запускаются одновременно.

**Проверьте:**
1. Откройте Google Apps Script
2. Нажмите на иконку **часов** (Triggers) слева
3. Проверьте список триггеров

**Должен быть ТОЛЬКО ОДИН триггер:**
- Function: `syncProgressToApp`
- Event source: Time-driven
- Type of time based trigger: Minutes timer
- Select minute interval: Every 15 minutes

**Удалите все лишние триггеры!**

### 2. **Тестирование скрипта**
Если вы много раз запускали скрипт вручную (кнопка "Run"), это также считается в лимит.

### 3. **Старые проекты**
Если у вас есть другие Google Apps Script проекты, они также используют общий лимит аккаунта.

---

## ✅ Решение

### Шаг 1: Удалите все лишние триггеры
1. Apps Script → **Triggers** (иконка часов слева)
2. Удалите все триггеры для `syncProgressToApp`
3. Создайте **ОДИН** новый триггер:
   - Function: `syncProgressToApp`
   - Event: Time-driven → Minutes timer → **Every 15 minutes**

### Шаг 2: Подождите до завтра
Лимиты сбрасываются каждые 24 часа (по времени Google, Pacific Time).

### Шаг 3: Оптимизируйте интервал
Если ошибка повторяется:
- Измените интервал с **каждые 15 минут** на **каждые 30 минут**
- Это уменьшит количество запросов в день с ~96 до ~48

### Шаг 4: Проверьте логи
В Apps Script → **Executions** (значок списка слева) проверьте:
- Сколько раз скрипт выполнялся сегодня
- Были ли ошибки

## 📈 Расчёт использования

**При интервале каждые 15 минут:**
- 4 запроса в час
- 96 запросов в сутки
- ✅ Укладываемся в лимит 20,000

**Вывод:** Проблема скорее всего в дублирующихся триггерах или тестовых запусках.

## 🔧 Дополнительная оптимизация

Если проблема сохраняется, можно добавить кэширование в скрипт:

```javascript
function syncProgressToApp() {
  const cache = CacheService.getScriptCache();
  const cacheKey = 'last_progress_data';
  
  // ... ваш код получения progressData ...
  
  const newData = JSON.stringify(progressData);
  const cachedData = cache.get(cacheKey);
  
  // Отправляем только если данные изменились
  if (newData === cachedData) {
    Logger.log('⏭️ Данные не изменились, пропускаем отправку');
    return;
  }
  
  // ... отправка на сервер ...
  
  // Сохраняем в кэш
  cache.put(cacheKey, newData, 21600); // 6 часов
}
```

Это уменьшит количество реальных HTTP-запросов, если проценты не изменились.
